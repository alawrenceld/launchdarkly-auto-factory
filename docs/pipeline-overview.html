AutoFactory — from pull request to guarded release

Two autonomous phases. Phase 1 turns a pull request into release-ready resources (flag, metrics, tests, review). Phase 2 notices the deploy and runs the release — guarded by the metrics Phase 1 created. Hover any node for the actual logic.

Phase 1 — PR → release-ready resources

Runs in CI on every pull request. Five LaunchDarkly-hosted agents, chained by an agent graph.

Step 1
Developer opens a PR
a plain feature PR — no flags, no ceremony
Nothing special required of the author. The change is written unconditionally (no flag in the diff). Everything that follows — flagging decision, wiring, metrics, tests — is the pipeline's job, not the developer's.
→
Step 2
Action invokes AutoFactory
GitHub Action on the PR checkout
A reusable CI action runs on pull_request with the PR branch checked out — that checkout is the agents' sandbox.
  • Resolves the agent graph + agent instructions live from LaunchDarkly (AI Configs); prompts are config, not code.
  • A provider flag picks the execution backend (e.g. Anthropic API vs. an internal agent runtime) per run.
→
Step 3
Research & Planner
read-only recon + classification
Read-only agent (diff, files, grep). Produces the brief every later agent consumes:
  • Classifies the PR: pr_type, risk_level, change_patterns, user-facing impact.
  • Detects the repo's existing flag/test/telemetry patterns so downstream agents match them.
  • Routing: tags flag_worthy; tagging skip_flagging=true short-circuits the whole chain (docs-only PRs, dep bumps…).
→
Step 4
Flag Implementer
creates the flag dark, wires the code
Creates a real boolean flag (targeting OFF — merge ≠ release) and rewires the PR:
  • New behavior goes behind the flag; the control path must preserve existing behavior exactly.
  • Writes a release manifest (.release-flags/pr-N.json) — the contract Phase 2 acts on.
  • Commits to the PR branch. Write access is a capability granted per graph edge (config, not code); idempotent on re-runs (existing flag → reuse).
→
Step 5
Metrics Author
how will we know it's safe?
Decides how the change is measured during the guarded release:
  • Covers the categories the change exposes: error (lower-better), latency (numeric ms, lower-better), business (higher-better).
  • Instruments track() events on the flagged path (fail-safe — telemetry can never break a request), then creates the metrics off those events.
  • Rule: metric randomization unit must match the flag's rollout unit, or the metric silently measures nothing.
  • Classifies each metric killswitch / pause / monitoring and writes metricKeys into the manifest — closing the loop to Phase 2.
→
Step 6
Flag Testing
flag-on + flag-off proof
Writes paired tests for every flagged path — one proving the new behavior (flag on), one proving the old behavior is untouched (flag off).
  • Matches the repo's existing test framework and conventions.
  • Runs the suite itself and fixes failures; only pushes once green.
→
Step 7
Code Reviewer
independent verdict + risk level
Read-only review of the whole enriched diff (author's change + all agent commits):
  • Checks logic, error handling, the flag wiring (control path intact?), the instrumentation (events match each metric's event_key), and test quality.
  • Emits APPROVE / REJECT + risk_level; conservative blocking — false-positive blocks erode trust.
  • The verdict + risk gate the configured approval mode (auto-apply / gate on risk / require a human).
→
Step 8
Human merges
merge ≠ release
The PR merges like any other — with flag, metrics, tests, and review attached. The feature ships dark: the flag is off, the control path serves, and the release manifest rides along in the repo waiting for Phase 2.

Phase 2 — deploy → guarded release (Beacon)

Event-driven release orchestrator. Hosted separately from the app — customers run nothing.

Step 9
CD deploys the services
normal pipeline, any provider
The merge deploys through whatever CD the team already uses (the demo uses Railway's GitHub auto-deploy). AutoFactory doesn't own the deploy — it only needs to hear about it.
→
Step 10
Deploy webhook → Beacon
open contract, thin adapters
Anything that can POST {service, sha} with the shared secret can announce a deploy: a provider's native webhook (via a small adapter that translates its payload), a CI step, or curl.
  • Only a successful deploy event acts; build/fail/crash events are acknowledged and ignored.
  • Deliveries are idempotent — providers retry, and a re-delivered event can't double-trigger a release.
→
Step 11
SHA diff → discovery
what's new since the last deploy?
Beacon diffs .release-flags/ between the deployed SHA and the previous one — a manifest present now but not before is a release to run.
  • previousSha priority: explicit value in the notification → Beacon's own deploy-state store (last SHA seen per service+environment) → none (first deploy: everything current is new).
  • The store keeps two SHAs deep so a re-delivered notification re-diffs the same range instead of the empty one.
  • Survives batched merges and redeploys; no dependency on the CD system knowing its own history.
→
Step 12
Scope & fullstack check
is every needed service deployed?
Each manifest carries a scope — frontend, backend, or fullstack — and each service registers which side it is.
  • Wrong side's notification → skip (the right side will handle it).
  • Fullstack: Beacon asks the other side's live status endpoint for its deployed SHA and checks that SHA contains the same manifest. Both sides present → release; otherwise wait — the other service's own deploy notification re-evaluates.
→
Step 13
Release trigger
method & shape by priority
One decision chain picks how the flag releases:
  • Priority: manifest releaseOverrides → the flag's configured release policy → defaults: guarded if metrics are attached, progressive if not (immediate only if explicitly requested).
  • Flags ship dark, so the trigger turns targeting on in the same atomic change that starts the release — no traffic moves except through the stages.
  • Guarded stage allocations are capped at 50% (the comparison needs a control group); the release completes to 100% after the final monitored stage.
→
Step 14
Guarded release runs
staged %, metrics watching
LaunchDarkly shifts traffic stage by stage (e.g. 20% → 50%) while the Phase-1 metrics compare treatment vs. control live on real traffic — no historical baseline needed.
  • Killswitch/pause metrics gate progression; auto-rollback is armed per metric.
  • The randomization unit from the manifest keeps measurement aligned with the rollout.
→
Step 15
Beacon monitors to the end
completed · reverted · stopped
Beacon resolves the release id and polls it to a terminal state, detached from the webhook request (a release can run for days):
  • completed — rolled out to 100%.
  • reverted — a guardrail regressed; LaunchDarkly flipped traffic back to control.
  • monitoring_stopped — a human intervened.
  • Monitoring failures never affect the release itself — it runs server-side regardless.
→
Safety net
Auto-rollback
regression → control, no human
If a killswitch metric regresses, the release reverts on its own — traffic returns to the control variation in seconds, without a deploy and without paging anyone. "Rollback" here means flipping the flag, not rolling back code: the bad path simply stops being served. The reverted release is a terminal state Beacon reports.
